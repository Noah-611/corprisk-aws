# CorpRisk 프로젝트 보고서

## 1. 프로젝트 주제

본 프로젝트의 주제는 **OpenDART 재무제표 데이터를 활용한 기업 재무 리스크 분석 웹서비스 구축**이다.

사용자가 웹 화면에서 기업명을 검색하면, 서버는 OpenDART API를 통해 해당 기업의 재무제표 데이터를 조회한다. 이후 주요 재무비율을 계산하고, 이를 바탕으로 기업의 재무 리스크 등급을 제공한다. 분석 결과는 웹 화면에서 확인할 수 있으며, 데이터베이스에 저장된 최근 분석 이력도 별도로 조회할 수 있다.

프로젝트 이름은 **CorpRisk**로 정했다. 기업을 의미하는 Corporation과 리스크 분석을 의미하는 Risk를 결합한 이름이다.

---

## 2. 개발 배경 및 목적

클라우드 컴퓨팅 수업에서는 EC2, S3, AWS CLI, Terraform, ALB, Auto Scaling Group, RDS 등 여러 AWS 서비스를 실습했다. 이번 프로젝트에서는 각 서비스를 따로 사용하는 데서 끝나지 않고, 실제로 동작하는 하나의 웹서비스 구조로 연결해보는 것을 목표로 했다.

처음에는 단순한 웹페이지를 만드는 것보다, 외부 데이터를 가져와 분석하고 결과를 저장하는 서비스가 클라우드 구성 요소를 설명하기에 더 적합하다고 판단했다. 그래서 금융 데이터 중 비교적 접근이 쉬운 OpenDART API를 사용했고, 기업의 재무제표 데이터를 기반으로 재무비율과 리스크 등급을 계산하는 서비스를 구현했다.

본 프로젝트의 목적은 다음과 같다.

* OpenDART API를 활용해 실제 기업 재무 데이터를 조회한다.
* 재무제표 데이터를 기반으로 주요 재무비율을 계산한다.
* 계산 결과를 웹 화면과 JSON API로 제공한다.
* 분석 결과를 데이터베이스에 저장하고 이력으로 관리한다.
* AWS EC2, ALB, ASG, RDS를 활용해 고가용성 구조로 배포한다.
* Terraform을 사용해 인프라 구성을 코드로 관리한다.

## 3. 주요 기능

### 3.1 기업 검색 및 자동완성

사용자는 메인 화면에서 기업명을 입력할 수 있다.
입력값을 기준으로 OpenDART 기업 코드 목록에서 관련 기업을 검색하고, 상장 기업을 중심으로 자동완성 목록을 제공한다.

예를 들어 `삼성`을 입력하면 삼성전자, 삼성전기, 삼성SDS 등 관련 기업이 표시된다. 사용자는 이 중 하나를 선택해 분석을 진행할 수 있다.

![기업 검색 자동완성](images/02-search-autocomplete.png)

---

### 3.2 OpenDART 재무제표 조회

사용자가 기업을 선택하면 서버는 해당 기업의 종목코드를 기준으로 OpenDART 기업 고유번호를 찾는다. 이후 OpenDART 재무제표 API를 호출하여 선택한 연도의 사업보고서 데이터를 조회한다.

OpenDART API는 일반 기업명이 아니라 기업 고유번호를 기준으로 데이터를 조회하기 때문에, 기업명 검색과 고유번호 매핑 과정이 필요했다. 이를 위해 OpenDART 기업 코드 XML 데이터를 내려받고, 종목코드와 기업 고유번호를 연결하는 로직을 구현했다.

---

### 3.3 재무비율 계산

조회한 재무 데이터를 기반으로 다음 재무비율을 계산했다.

| 지표     | 계산 의미           |
| ------ | --------------- |
| 부채비율   | 자기자본 대비 부채 규모   |
| 유동비율   | 단기 부채 상환 여력     |
| 영업이익률  | 매출액 대비 영업이익 비율  |
| 순이익률   | 매출액 대비 당기순이익 비율 |
| 자기자본비율 | 총자산 중 자기자본 비중   |

계산된 지표는 결과 화면에 표 형태로 표시된다.

![분석 결과](images/03-analysis-result.png)

---

### 3.4 리스크 점수 및 등급 산출

계산된 재무비율을 기준으로 규칙 기반 리스크 점수를 산출했다.
부채비율이 높거나 유동비율이 낮은 경우 리스크 점수가 높아지도록 했고, 수익성 지표가 낮은 경우에도 리스크 판단에 반영했다.

최종 리스크 등급은 낮음, 중간, 높음으로 구분했다.
이 등급은 투자 추천을 위한 판단 기준이 아니라, 재무제표 데이터를 이해하기 쉽게 표현하기 위한 교육용 지표이다.

---

### 3.5 분석 연도 선택

결과 화면에서는 분석 연도를 드롭다운으로 선택할 수 있도록 구현했다.
사용자가 연도를 변경하면 해당 연도의 재무제표 데이터를 다시 조회하고, 선택한 연도 기준으로 재무비율과 리스크 등급을 다시 계산한다.

![분석 연도 선택](images/04-year-selector.png)

---

### 3.6 과거 재무 추이 그래프

단일 연도 분석만으로는 기업의 흐름을 확인하기 어렵기 때문에, 여러 연도의 데이터를 조회해 과거 재무 추이를 그래프로 볼 수 있도록 했다.

그래프에서는 매출액, 영업이익, 당기순이익, 부채비율, 유동비율, 자기자본비율, 수익률, 성장률 등을 선택해서 확인할 수 있다. 이를 통해 특정 기업의 재무 상태가 과거에 비해 개선되고 있는지 또는 악화되고 있는지를 한 화면에서 비교할 수 있다.

![과거 재무 추이 그래프](images/05-history-chart.png)

---

### 3.7 분석 결과 저장 및 이력 조회

분석 결과는 데이터베이스에 저장된다.
로컬 개발 환경에서는 SQLite를 사용했고, AWS 배포 환경에서는 RDS MySQL을 사용했다.

분석 이력은 `/history` 페이지에서 확인할 수 있다. 이 기능을 통해 이전에 분석한 기업과 연도, 주요 재무비율, 리스크 등급을 다시 확인할 수 있다.

![분석 이력](images/06-history-page.png)

## 4. 시스템 구조

본 프로젝트의 전체 구조는 다음과 같다.

```text
User
  ↓
Application Load Balancer
  ↓
EC2 Auto Scaling Group
  ↓
FastAPI Application
  ↓
Amazon RDS MySQL
```

애플리케이션은 FastAPI로 구현했고, 화면 렌더링에는 Jinja2 템플릿을 사용했다.
정적 파일은 CSS와 JavaScript로 구성했으며, 그래프 출력에는 Chart.js를 사용했다.

AWS 배포 환경에서는 ALB 뒤에 EC2 인스턴스 2대를 두고, 두 인스턴스가 동일한 FastAPI 애플리케이션을 실행하도록 구성했다. 분석 결과는 공통 RDS MySQL에 저장되므로, 어느 인스턴스로 요청이 전달되어도 같은 분석 이력을 조회할 수 있다.

## 6. AWS 구성

### 6.1 EC2

FastAPI 애플리케이션을 실행하는 서버로 EC2를 사용했다.
EC2 인스턴스가 생성될 때 user data 스크립트를 통해 GitHub 저장소를 clone하고, Python 가상환경 생성, 패키지 설치, 환경변수 설정, systemd 서비스 등록을 자동으로 수행하도록 했다.

이를 통해 인스턴스를 새로 생성하더라도 수동으로 서버를 설정하지 않고 애플리케이션이 실행되도록 구성했다.

---

### 6.2 Application Load Balancer

외부 사용자의 요청은 Application Load Balancer를 통해 EC2 인스턴스로 전달된다.
ALB는 Target Group에 등록된 EC2 인스턴스의 `/health` 경로를 확인하고, 정상 상태의 인스턴스에만 트래픽을 전달한다.

![Target Group Healthy](images/07-target-group-healthy.png)

---

### 6.3 Auto Scaling Group

Auto Scaling Group은 EC2 인스턴스 2대를 유지하도록 구성했다.
프로젝트 규모상 실제 트래픽 기반 확장보다는, 인스턴스 장애나 교체 상황에서 서비스가 계속 유지되는 구조를 확인하는 데 중점을 두었다.

한 인스턴스를 종료해도 ASG가 새 인스턴스를 생성하고, ALB가 정상 인스턴스에만 요청을 전달하는 것을 확인했다.

![Auto Scaling Group](images/08-asg-instances.png)

---

### 6.4 RDS MySQL

분석 결과 이력 저장을 위해 Amazon RDS MySQL을 사용했다.
로컬 개발 단계에서는 SQLite를 사용했지만, AWS 배포 단계에서는 RDS 연결 문자열을 환경변수로 주입하여 RDS에 분석 결과가 저장되도록 했다.

RDS는 외부 공개 접근을 비활성화하고, 애플리케이션이 실행되는 EC2에서만 접근할 수 있도록 구성했다.

![RDS](images/09-rds.png)

---

### 6.5 S3

S3는 프로젝트 산출물 또는 향후 분석 리포트 저장을 위한 버킷으로 구성했다.
현재 서비스에서는 분석 결과를 RDS에 저장하고 있지만, 향후 CSV 또는 PDF 형태의 리포트를 생성해 S3에 저장하는 방식으로 확장할 수 있다.

---

### 6.6 Terraform

AWS 인프라는 Terraform으로 구성했다.
EC2, Security Group, Launch Template, Auto Scaling Group, ALB, Target Group, RDS, S3 관련 리소스를 코드로 작성했다.

Terraform을 사용한 이유는 콘솔에서 직접 만든 리소스보다 재현성과 관리 측면에서 유리하기 때문이다. 또한 인프라 변경 내역을 코드로 확인할 수 있어 프로젝트 제출 시 구성 과정을 설명하기에도 적합했다.

## 9. 구현 과정에서 해결한 문제

### 9.1 OpenDART 기업명 조회 문제

OpenDART API는 기업명을 직접 넣어 재무제표를 조회하는 방식이 아니라, 기업 고유번호를 기반으로 조회한다.
처음에는 기업명만으로 데이터를 가져오려 했지만 원하는 결과가 나오지 않았다.

이를 해결하기 위해 OpenDART 기업 코드 목록을 내려받고, 종목코드와 기업 고유번호를 매핑하는 방식으로 수정했다. 이후 사용자는 기업명을 입력하지만, 내부적으로는 종목코드와 기업 고유번호를 이용해 정확한 기업 데이터를 조회하도록 처리했다.

---

### 9.2 EC2 배포 자동화 문제

EC2 인스턴스가 새로 생성될 때마다 수동으로 Git clone, 패키지 설치, 환경변수 설정, 서버 실행을 반복하는 것은 비효율적이었다.

이를 해결하기 위해 Launch Template의 user data에 배포 과정을 작성했다.
인스턴스가 시작되면 자동으로 저장소를 clone하고, Python 가상환경을 만든 뒤, systemd 서비스로 FastAPI 애플리케이션을 실행하도록 구성했다.

---

### 9.3 GitHub URL 오타 문제

초기 배포 과정에서 user data에 전달되는 GitHub 저장소 URL에 오타가 있어 EC2가 애플리케이션 코드를 정상적으로 clone하지 못했다.
해당 문제는 EC2 console output을 확인해 원인을 찾았고, Terraform 변수 값을 수정한 뒤 다시 배포하여 해결했다.

---

### 9.4 Python 버전 차이 문제

로컬 개발 환경은 Python 3.12였지만, EC2 배포 환경에서는 Python 3.9가 사용되었다.
이 차이로 인해 Python 3.10 이상에서 지원하는 타입 힌트 문법이 EC2에서 오류를 발생시켰다.

문제 확인 후 `str | None` 형태의 타입 힌트를 `Optional[str]` 형태로 수정하여 Python 3.9 환경에서도 애플리케이션이 실행되도록 했다.

---

### 9.5 ALB Health Check 문제

애플리케이션이 실행되지 않거나 health check를 통과하지 못하면 Target Group에서 unhealthy 상태가 발생했다.
이를 확인하기 위해 EC2 console output과 systemd 로그를 확인했고, `/health` 엔드포인트를 기준으로 ALB가 정상 상태를 판단하도록 구성했다.

최종적으로 EC2 2대가 모두 healthy 상태가 되는 것을 확인했다.

## 10. 한계점 및 개선 방향

본 프로젝트는 수업 과제 범위 안에서 구현했기 때문에 몇 가지 한계가 있다.

첫째, 리스크 등급은 규칙 기반으로 계산된다.
실제 기업 분석에서는 산업별 평균, 기업 규모, 현금흐름, 부채 만기 구조 등 더 많은 요소가 필요하다. 향후에는 산업군별 기준을 반영하거나, 과거 부실 기업 데이터를 활용해 더 정교한 위험 평가 방식으로 개선할 수 있다.

둘째, OpenDART 데이터는 기업별 공시 여부와 시점에 따라 조회 가능한 연도가 다르다.
현재는 사용자가 연도를 선택할 수 있도록 구현했지만, 향후에는 실제 조회 가능한 연도만 자동으로 표시하도록 개선할 수 있다.

셋째, 현재는 단일 기업 분석 중심이다.
향후에는 여러 기업을 선택해 같은 지표를 비교하거나, 업종 평균과 비교하는 기능을 추가할 수 있다.

넷째, S3 활용 범위가 제한적이다.
현재는 프로젝트 산출물 또는 리포트 저장을 위한 구성에 가깝지만, 향후 분석 결과를 CSV, PDF 형태로 생성해 S3에 저장하고 다운로드하는 기능으로 확장할 수 있다.

## 11. 결론

이번 프로젝트를 통해 단순히 EC2 서버를 실행하는 수준을 넘어, ALB, Auto Scaling Group, RDS, Terraform을 함께 사용하는 웹서비스 구조를 직접 구성할 수 있었다.

특히 FastAPI 애플리케이션을 EC2 2대에 배포하고, ALB를 통해 정상 인스턴스에만 요청이 전달되도록 구성한 점이 핵심이었다. 또한 분석 결과를 RDS MySQL에 저장하여 여러 인스턴스가 동일한 데이터를 공유하는 구조를 구현했다.

개발 과정에서는 OpenDART 기업 코드 매핑, EC2 자동 배포, Python 버전 차이, ALB Health Check 문제 등을 해결했다. 이 과정에서 로컬 개발 환경과 실제 클라우드 배포 환경이 다를 수 있다는 점을 경험했고, 로그를 기반으로 문제를 추적하는 과정의 중요성을 확인할 수 있었다.

최종적으로 CorpRisk는 OpenDART API 기반 재무 데이터 수집, 재무비율 계산, 리스크 등급 산출, 분석 결과 저장, AWS 고가용성 배포 구조를 포함한 클라우드 기반 웹서비스로 구현되었다.
